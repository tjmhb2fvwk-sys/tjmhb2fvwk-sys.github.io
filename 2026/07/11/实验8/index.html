实验报告8：Web软件部署

实验报告：Web软件部署

1、实验目的

综合使用所学各类工具，构建和模拟一个网络环境，体验和实现 Web 静态站点从开发到部署的全过程。

2、实验环境

  • 操作系统：Windows 11 专业版 22H2 + Ubuntu Server 22.04（VirtualBox 虚拟机）
  • 开发环境：Node.js v20.10.0，Hexo CLI v7.3.0
  • 虚拟机配置：node1（192.168.56.102），内存 2GB，硬盘 20GB
  • 所需账号：GitHub 账号

3、实验过程

步骤一：确认 Hexo 博客正常运行

在本地 Windows 环境中进入博客项目目录，执行 hexo clean 清理缓存，hexo generate 生成静态文件，hexo server 启动本地预览服务，验证博客可正常访问。


图1：Hexo 生成静态文件并启动本地预览

步骤二：创建 GitHub 仓库

登录 GitHub 账号，创建两个仓库：源码仓库 my-blog（Private，存放 Hexo 源码）和部署仓库 用户名.github.io（Public，托管静态页面）。

GitHub源码仓库和部署仓库创建完成
图2：GitHub 源码仓库和部署仓库创建完成

步骤三：配置 GitHub Actions 自动化部署

在源码仓库中创建 .github/workflows/deploy.yml 文件，配置 GitHub Actions 工作流：检出代码 → 安装 Node.js → 缓存依赖 → 安装依赖 → 生成静态文件 → 部署到 GitHub Pages。生成 Personal Access Token（勾选 repo 权限），在源码仓库 Settings → Secrets 中添加 PERSONAL_TOKEN。

GitHub Actions 工作流配置
图3：.github/workflows/deploy.yml 工作流配置


图4：GitHub Secrets 中配置 PERSONAL_TOKEN

推送代码到 GitHub 后，GitHub Actions 自动触发构建和部署。在部署仓库中启用 GitHub Pages，访问 https://用户名.github.io 查看部署结果。


图6：GitHub Pages 部署成功后的博客页面

步骤四：使用 scp 部署到虚拟机（备选）

在虚拟机 node1 上安装 Nginx：sudo apt install nginx -y，创建站点目录 /var/www/blog，配置 Nginx 站点并启用。在本地生成静态文件后，使用 scp 同步到虚拟机。


图7：Nginx 站点配置文件


图8：使用 scp 同步静态文件到虚拟机

访问 http://192.168.56.102 查看部署结果。


图9：虚拟机 Nginx 部署的博客页面

4、实验结果

测试项 目标 结果
Hexo 本地构建 hexo generate ✅ 成功
GitHub Actions 自动部署流水线 ✅ 成功
GitHub Pages https://用户名.github.io ✅ 可访问
Nginx 部署 http://192.168.56.102 ✅ 可访问

5、知识总结

阶段 工具/平台 作用
开发阶段 VS Code + Markdown 编写博客内容
构建阶段 Hexo CLI Markdown → 静态 HTML
CI/CD GitHub Actions 自动构建和部署
生产环境 GitHub Pages 托管静态网站
备选部署 Nginx + rsync 服务器部署

6、出现问题

问题1：GitHub Actions 部署失败。

  • 原因：Personal Access Token 权限不足，未勾选 repo 权限。
  • 解决：重新生成 Token，确保勾选 repo 完整权限，在 Secrets 中更新。

问题2：部署后 GitHub Pages 显示 404。

  • 原因：仓库名称不是 用户名.github.io。
  • 解决：确保部署仓库命名为 用户名.github.io。

问题3：部署后页面样式丢失。

  • 原因：站点 URL 配置错误。
  • 解决：检查 _config.yml 中的 url 字段，设置为 https://用户名.github.io。

7、心得体会
