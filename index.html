实验报告1：Web开发环境

实验报告：Web开发环境

1、实验目的

安装 Web 开发环境，构建一个静态站点。了解和实践 Web 服务器、软件包及管理器等工具和概念。

2、实验环境

  • 操作系统：Windows 11 专业版 22H2
  • 软件版本：Node.js v24.18.0，npm v11.16.0，Hexo CLI v4.3.2
  • 运行环境：Node.js 运行时，npm 包管理器

3、实验过程

步骤一：安装 Node.js 与 npm

访问 Node.js 官网（https://nodejs.org/），下载 LTS 版本的 Windows 安装包并运行安装。安装过程中务必勾选 “Automatically install the necessary tools” 和 “Add to PATH” 选项。安装完成后，在命令提示符中执行 node -v 和 npm -v 验证安装。


图1：Node.js 安装完成界面


图2：验证 Node.js 和 npm 安装版本

步骤二：配置 npm 镜像源

使用 npm config set registry https://registry.npmmirror.com 切换到国内镜像源，加速依赖下载。执行 npm config get registry 验证配置是否生效。

步骤三：安装 Hexo CLI 并初始化博客

执行 npm install -g hexo-cli 全局安装 Hexo CLI，使用 hexo -v 验证安装。然后在 D:\Blog 目录下执行 hexo init my-blog 初始化博客项目，并进入项目目录执行 npm install 安装项目依赖。


图4：Hexo CLI 版本验证


图5：Hexo 博客项目初始化过程

步骤四：本地预览博客

在项目目录下执行 hexo server 启动本地预览服务，打开浏览器访问 http://localhost:4000 查看默认博客首页。


图6：Hexo 博客本地预览首页

4、实验结果

测试项 目标 结果
Node.js 安装 node -v ✅ v20.10.0
npm 安装 npm -v ✅ 10.2.3
Hexo CLI 安装 hexo -v ✅ v7.3.0
博客初始化 hexo init ✅ 成功
本地预览 http://localhost:4000 ✅ 成功

5、知识总结

命令 作用
node -v 查看 Node.js 版本
npm -v 查看 npm 版本
npm install -g 包名 全局安装软件包
hexo init 项目名 初始化 Hexo 博客项目
hexo server 启动本地预览服务

6、出现问题

问题1：执行 npm install 时速度极慢。

  • 原因：默认使用国外 npm 镜像源。
  • 解决：切换到国内镜像源 npm config set registry https://registry.npmmirror.com。

问题2：hexo init 速度极慢或失败。

  • 原因：网络问题，无法访问 GitHub。
  • 解决：配置代理，或使用 Gitee 镜像源。

7、心得体会

通过本次实验，我掌握了 Node.js 和 npm 的安装与配置方法，理解了 Web 开发环境的基本组成，包括运行时环境和包管理器。Hexo 作为静态站点生成器，让我体验到了快速构建博客的便捷流程。同时，我也了解到切换国内镜像源对提升开发效率的重要作用。在后续的学习中，我将继续探索 Hexo 的主题定制和更多高级功能。【课程大纲-思政融合点1】

实验报告2：代码编辑与管理

实验报告：代码编辑与管理

1、实验目的

安装和使用文本代码编辑器完成 Markdown 的编辑和渲染，使用 Git 进行版本管理等。了解和实践代码编辑器的配置、Markdown 语法、版本管理工具的使用等。

2、实验环境

  • 操作系统：Windows 11 专业版 22H2
  • 软件版本：Git 2.55.0，VS Code 1.128.0

3、实验过程

步骤一：安装 Git for Windows

访问 Git 官网（https://git-scm.com/download/win）下载并运行安装包。在 “Select Components” 页面勾选 “Git Bash Here” 和 “Git GUI Here”，在 “Adjusting your PATH environment” 页面选择 “Git from the command line and also from 3rd-party software”。安装完成后重启命令提示符，执行 git --version 验证安装。

图3：VS Code 安装完成界面

图6：Markdown 源码与预览分屏模式

实验报告3：虚拟机安装与配置

实验报告：虚拟机安装与配置

1、实验目的

使用虚拟机工具安装 Linux 系统，了解和实践虚拟机的基本配置和网络模式配置。

2、实验环境

  • 操作系统：Windows 11
  • 软件版本：VirtualBox 7.0，Ubuntu Server 22.04 LTS
  • 虚拟机配置：内存 2GB，硬盘 20GB，双网卡（NAT + Host-Only）
  • 网络拓扑：Host-Only 网络，IP 段 192.168.56.0/24

3、实验过程

步骤一：下载与安装 VirtualBox

访问 VirtualBox 官网（https://www.virtualbox.org/）下载 Windows 版本安装包并安装，安装完成后重启电脑。

步骤二：下载 Ubuntu Server ISO 镜像

访问 Ubuntu 官网（https://ubuntu.com/download/server）下载 22.04 LTS 版本的 ISO 镜像。

步骤三：创建第一台虚拟机（模板机）

打开 VirtualBox 点击 “新建”，配置虚拟机名称为 ubuntu-template，类型为 Linux 64-bit，ISO 映像选择下载的 Ubuntu Server ISO，内存分配 2048 MB，硬盘为 VDI 格式动态分配 20 GB。配置虚拟机网络：适配器1为 NAT，适配器2为仅主机（Host-Only）适配器。


图1：VirtualBox 新建虚拟机基本配置


图2：虚拟机 NAT + Host-Only 双网卡配置

步骤四：安装 Ubuntu Server

启动虚拟机，选择 “Try or Install Ubuntu Server” 开始安装。语言和键盘布局保持默认，网络配置保持 DHCP，存储选择 “Use an entire disk”。设置用户名和密码，务必按空格键选中 “Install OpenSSH server”。等待安装完成后重启。

步骤五：配置静态 IP 地址

登录系统后编辑 /etc/netplan/50-cloud-init.yaml，为 enp0s8（仅主机网卡）配置静态 IP 地址 192.168.56.101/24。执行 sudo netplan apply 使配置生效，使用 ip addr show 验证 IP 配置。


图3：Netplan 网络配置文件内容


图4：IP 地址配置验证

步骤六：克隆虚拟机

关闭模板机后，在 VirtualBox 中右键点击模板机选择 “克隆”，选择 “完全克隆” 方式，分别克隆出 node1、node2、node3。


图5：VirtualBox 完全克隆操作界面

步骤七：修改克隆机的主机名和 IP

对每台克隆机执行 sudo hostnamectl set-hostname node1 修改主机名，编辑 /etc/netplan/50-cloud-init.yaml 修改 IP 地址（node1→102，node2→103，node3→104），执行 sudo netplan apply 使配置生效。

4、实验结果

虚拟机 主机名 IP地址 状态
模板机 template 192.168.56.101 ✅ 已配置
节点1 node1 192.168.56.102 ✅ 已克隆
节点2 node2 192.168.56.103 ✅ 已克隆
节点3 node3 192.168.56.104 ✅ 已克隆

5、知识总结

网络模式 特点 用途
NAT 通过宿主机上网，外部不可直达 日常上网
桥接 直接接入宿主机局域网 被外部访问
仅主机 宿主机与虚拟机形成隔离私网 内网互连

6、出现问题

问题1：创建虚拟机时提示 “VT-x/AMD-V 未启用”。

  • 原因：BIOS 中未开启硬件虚拟化。
  • 解决：重启电脑进入 BIOS，启用 VT-x/AMD-V 虚拟化技术。

问题2：克隆后网络无法使用。

  • 原因：克隆时保留了原网卡的 MAC 地址。
  • 解决：在 VirtualBox 中对克隆机重新生成 MAC 地址。

7、心得体会

通过本次实验，我深入理解了虚拟化技术的概念和工作原理，掌握了 VirtualBox 的安装、虚拟机创建和克隆操作。双网卡配置让我认识到不同网络模式的应用场景：NAT 用于上网，Host-Only 用于内网互连。这种多机虚拟化环境为后续的 Linux 操作、SSH 远程管理等实验提供了基础设施。在配置静态 IP 时，我体会到了 Linux 网络配置的灵活性，也加深了对 Netplan 工具的理解。【课程大纲-思政融合点1】

实验报告4：Linux基本操作

实验报告：Linux基本操作

1、实验目的

使用命令行进行文件导航、软件包安装等，了解和实践命令行的基本使用。

2、实验环境

  • 操作系统：Ubuntu Server 22.04 LTS（VirtualBox 虚拟机）
  • 虚拟机配置：node1（192.168.56.102），内存 2GB，硬盘 20GB
  • 前置条件：已完成实验三，虚拟机网络配置正常

3、实验过程

步骤一：登录与查看系统信息

登录 Ubuntu Server 后，依次执行 whoami 查看当前用户、uname -a 查看内核与架构信息、lsb_release -a 查看 Ubuntu 版本、hostname 查看主机名、pwd 查看当前工作目录。

步骤二：文件与目录的基本操作

执行 ls -lh 列出当前目录文件的详细信息，使用 cd ~ 切换到用户主目录。使用 mkdir test_dir 创建新目录，touch test.txt 创建空文件，cp test.txt test_copy.txt 复制文件，mv test.txt /tmp/ 移动文件，rm -f test.txt 删除文件。

⚠️ rm -rf 命令会永久删除文件且不可恢复，请谨慎使用！

步骤三：查看系统资源状态

使用 free -h 查看系统内存使用情况，df -h 查看磁盘使用情况，top 查看进程和资源占用（按 q 退出），du -sh * 查看当前目录下各子目录的大小。

步骤四：软件包管理

执行 sudo apt update 更新软件包列表，sudo apt install nginx -y 安装 Nginx 软件包，sudo apt remove nginx -y 删除已安装的软件包，sudo apt upgrade -y 升级所有已安装的软件包。

步骤五：配置主机名与 hosts 解析

使用 sudo hostnamectl set-hostname template 修改主机名为 template，编辑 /etc/hosts 文件添加各节点的 IP 与主机名映射关系。


图1：/etc/hosts 主机名解析配置

步骤六：验证多机环境

在 node1 上执行 ping -c 3 8.8.8.8 测试外网连通性，ping -c 3 192.168.56.1 测试宿主机连通性，ping -c 3 192.168.56.102 测试 node2 连通性，ping -c 3 node3 测试主机名解析。


图2：多机环境网络连通性测试

4、实验结果

测试项 目标 结果
外网连通 8.8.8.8 ✅ 成功
宿主机连通 192.168.56.1 ✅ 成功
虚拟机互连 192.168.56.102 ✅ 成功
主机名解析 node3 ✅ 成功

5、知识总结

命令 作用 示例

6、出现问题

  • 原因：node3 未开机或 IP 地址配置错误。

实验报告5：Linux账户与权限管理

实验报告：Linux账户与权限管理

1、实验目的

使用命令行账户设置、权限管理等操作，了解和实践权限方面的操作和配置。

2、实验环境

  • 操作系统：Ubuntu Server 22.04 LTS（VirtualBox 虚拟机）
  • 虚拟机配置：template（192.168.56.101），内存 2GB，硬盘 20GB
  • 前置条件：已完成实验三，虚拟机网络配置正常

3、实验过程

步骤一：查看用户与组配置文件

使用 cat /etc/passwd 查看系统用户账户信息，sudo cat /etc/shadow 查看加密的密码信息（仅 root 可读），cat /etc/group 查看用户组信息，id 查看当前用户的 ID 信息。/etc/passwd 文件每行格式为：用户名:密码占位符:UID:GID:用户描述:主目录:登录Shell。


图1：/etc/passwd 用户账户信息

步骤二：创建和管理用户

使用 sudo adduser testuser 创建新用户，按提示设置密码并填写可选信息（直接按 Enter 跳过）。使用 sudo passwd testuser 修改用户密码，sudo usermod -aG sudo testuser 将用户加入 sudo 组赋予管理员权限。使用 groups testuser 查看用户所属组，su - testuser 切换用户，sudo deluser -r testuser 删除用户及其主目录。


图2：创建新用户 testuser 及 groups 验证

步骤三：查看用户主目录

使用 ls -la /home/testuser 查看新建用户的主目录内容，ls -la /etc/skel/ 查看新用户默认配置文件的模板目录。


图3：testuser 主目录默认文件

步骤四：理解权限格式

使用 ls -l 查看文件权限，权限字符串 -rwxr-xr-- 中：第一位表示文件类型（-为普通文件），第 2-4 位为所有者权限，第 5-7 位为组权限，第 8-10 位为其他人权限。权限数字对照表：7=rwx（读写执行），6=rw-（读写），5=r-x（读执行），4=r–（只读），0=—（无权限）。

步骤五：修改文件权限

创建测试文件后，使用 chmod 755 test.txt 数字法设置权限为 rwxr-xr-x，使用 chmod u+x test.txt 符号法为所有者增加执行权限，chmod g-w test.txt 去掉组的写权限，chmod o+r test.txt 为其他人增加读权限。


图4：chmod 权限修改前后对比

步骤六：修改文件所有者和组

使用 sudo chown root:root test.txt 同时修改文件所有者和组为 root，sudo chgrp www-data test.txt 仅修改所属组为 www-data，sudo chown dev test.txt 仅修改所有者为 dev。


图5：chown 修改文件所有者前后对比

4、实验结果

操作 命令 结果
创建用户 sudo adduser testuser ✅ 成功
赋予 sudo 权限 sudo usermod -aG sudo testuser ✅ 成功
权限修改 chmod 755 test.txt ✅ 成功
所有者修改 sudo chown root:root test.txt ✅ 成功

5、知识总结

数字 权限 含义
7 rwx 读+写+执行
6 rw- 读+写
5 r-x 读+执行
4 r– 只读
命令 作用
chmod 修改文件权限
chown 修改文件所有者
chgrp 修改文件所属组
adduser 创建新用户
usermod -aG 将用户加入组
deluser -r 删除用户及主目录

6、出现问题

问题1：执行 su - testuser 切换用户时提示 “user testuser does not exist or the user entry does not contain all the required fields”。

  • 原因：adduser 创建用户时由于系统异常导致 /etc/passwd 中用户记录写入不完整。
  • 解决：执行 sudo deluser --remove-home testuser 删除不完整用户，重新执行 sudo adduser testuser 成功创建。

问题2：sudo: command not found。

  • 原因：用户不在 sudo 组中。
  • 解决：以 root 用户登录，使用 sudo usermod -aG sudo testuser 将用户加入 sudo 组。

7、心得体会

通过本次实验，我掌握了 Linux 用户和组的管理方法，理解了 /etc/passwd、/etc/shadow 和 /etc/group 等关键系统配置文件的作用。文件权限的数字法和符号法修改让我对 Linux 权限模型有了直观认识。在学习过程中，我体会到 Linux 严格的权限体系对系统安全的重要性。作为未来的软件工程师，在生产环境中配置系统权限时必须谨慎，遵循最小权限原则。【课程大纲-思政融合点1】

实验报告6：Linux环境配置

实验报告：Linux环境配置

1、实验目的

使用命令行对系统进行个性化配置，了解和实践各类常用系统配置文件、用户配置文件、环境变量等的管理。

2、实验环境

  • 操作系统：Ubuntu Server 22.04 LTS（VirtualBox 虚拟机）
  • 虚拟机配置：template（192.168.56.101），内存 2GB，硬盘 20GB
  • 前置条件：已完成实验三，虚拟机可正常登录

3、实验过程

步骤一：查看环境变量

使用 env 查看当前所有环境变量，echo $PATH 查看可执行文件的搜索路径，echo $HOME 查看当前用户主目录，echo $USER 查看当前用户名。

步骤二：临时设置环境变量

使用 export PATH=$PATH:/home/dev/mybin 临时在 PATH 末尾添加自定义目录，export MY_VAR="Hello World" 临时创建自定义变量，echo $MY_VAR 验证变量值，unset MY_VAR 删除变量。

步骤三：永久设置环境变量（用户级）

编辑 ~/.bashrc 文件，在末尾添加 export PATH=$PATH:/home/dev/mybin、export MY_CUSTOM_VAR="自定义变量" 和 alias ll='ls -alF'。保存后执行 source ~/.bashrc 使配置生效。~/.bashrc 在每次打开新终端时执行，作用于当前用户。


图1：~/.bashrc 用户级环境变量配置

步骤四：永久设置环境变量（系统级）

使用 /etc/profile.d/ 目录创建 myenv.sh 文件设置系统级环境变量，这是最推荐的方式。执行 sudo nano /etc/profile.d/myenv.sh 创建文件，添加 export MY_GLOBAL_VAR="Hello from system profile"。保存后执行 source /etc/profile.d/myenv.sh 使配置立即生效，使用 env | grep MY 验证配置。


图2：/etc/profile.d/myenv.sh 系统级环境变量配置


图3：系统级环境变量验证结果

4、实验结果

配置文件 作用范围 验证结果
~/.bashrc 当前用户 ✅ 已配置别名和环境变量
/etc/profile.d/myenv.sh 所有用户 ✅ MY_GLOBAL_VAR 已生效

5、知识总结

配置文件 加载时机 作用范围
~/.bashrc 每次打开新终端 当前用户
~/.profile 用户登录时 当前用户
/etc/profile 用户登录时 所有用户
/etc/environment 系统启动时 所有用户
/etc/profile.d/*.sh 用户登录时 所有用户

6、出现问题

问题1：修改 ~/.bashrc 后配置不生效。

  • 原因：未执行 source ~/.bashrc 重新加载配置文件。
  • 解决：执行 source ~/.bashrc 或在当前会话中手动执行新增的 export 命令。

问题2：env | grep MY 没有输出。

  • 原因：环境变量未正确生效，可能是 export 命令语法错误（等号两边有空格）或使用了中文引号。
  • 解决：检查 /etc/profile.d/myenv.sh 文件内容，确保格式正确（export VAR="value"，等号两边无空格），重新执行 source。

7、心得体会

通过本次实验，我理解了 Linux 环境变量的作用域和配置方法，区分了用户级配置（~/.bashrc）和系统级配置（/etc/profile.d/）的加载时机和影响范围。alias 别名功能让我体会到了命令行个性化和效率提升的方法。在配置环境变量的过程中，我认识到配置文件加载顺序的重要性，也加深了对 Linux Shell 启动过程的理解。系统级环境变量的配置需要 sudo 权限，体现了系统管理员的职责边界。【课程大纲-思政融合点1】

实验报告7：远程管理

实验报告：远程管理

1、实验目的

配置和使用 SSH 等工具进行远程登录、文件传输等操作，体验和完成基本的远程管理任务。

2、实验环境

  • 操作系统：Ubuntu Server 22.04 LTS（VirtualBox 虚拟机）
  • 虚拟机配置：node1（192.168.56.102）、node2（192.168.56.103）、node3（192.168.56.104）
  • 网络拓扑：Host-Only 网络，IP 段 192.168.56.0/24
  • 前置条件：已完成实验三，两台虚拟机网络互通且 SSH 服务已安装

3、实验过程

步骤一：检查并安装 SSH 服务

在所有虚拟机上执行 sudo systemctl status ssh 检查 SSH 服务状态。如未安装，执行 sudo apt install openssh-server -y 安装，使用 sudo systemctl enable ssh 设置开机自启。

步骤二：配置 SSH 服务器

编辑 /etc/ssh/sshd_config，确认 PermitRootLogin no（禁止 root 直接登录）、PubkeyAuthentication yes（允许公钥认证）、PasswordAuthentication yes（允许密码认证）。修改完成后执行 sudo systemctl restart ssh 重启服务。

步骤三：生成 SSH 密钥对（在 node1 上）

在 node1 上执行 ssh-keygen -t ed25519 -C "node1_key" 生成 ED25519 算法的密钥对，全部按 Enter 使用默认值。生成后使用 ls ~/.ssh/ 查看密钥文件（id_ed25519 私钥和 id_ed25519.pub 公钥），使用 cat ~/.ssh/id_ed25519.pub 查看公钥内容。

步骤四：将公钥复制到远程服务器

在 node1 上执行 ssh-copy-id tang@192.168.56.103，首次连接需输入 yes 确认主机指纹，然后输入 node2 上 tang 用户的密码。公钥会自动复制到 node2 的 ~/.ssh/authorized_keys 文件中。


图4：ssh-copy-id 复制公钥到 node2

步骤五：测试免密登录

在 node1 上执行 ssh tang@192.168.56.103，应无需密码直接登录。登录后执行 hostname 确认当前在 node2 上，执行 exit 退出。


图5：从 node1 免密登录 node2

步骤六：使用 SCP 传输文件

在 node1 上创建测试文件并上传到 node2，然后从 node2 下载文件到 node1。

SCP 上传和下载成功
图6：SCP 上传和下载文件成功

步骤七：使用 rsync 同步文件

在 node1 上创建测试目录结构和文件，使用 rsync 进行本地同步、远程推送和远程拉取操作。

rsync同步过程截图（显示增量传输的信息）

4、实验结果

测试项 目标 结果

SSH 免密登录 node1 → node2 ✅ 成功

5、知识总结

命令 作用

scp 源 目标 基于 SSH 的加密文件传输

6、出现问题

问题1：执行 sudo systemctl status ssh 提示 “Unit ssh.service could not be found”。

  • 原因：安装 Ubuntu Server 时未勾选 “Install OpenSSH server”。
  • 解决：执行 sudo apt install openssh-server -y 安装 SSH 服务。

问题2：首次 SSH 连接时提示主机认证信息确认。

  • 原因：首次连接未知主机，SSH 需要确认主机指纹。
  • 解决：输入 yes 确认，该信息会保存在 ~/.ssh/known_hosts 中。

7、心得体会

通过本次实验，我深入理解了 SSH 协议的原理和工作方式，掌握了密钥认证相比密码认证在安全性上的优势。免密登录的配置让我体会到了自动化运维的便利性，SCP 和 rsync 的实践加深了我对加密传输和增量同步的理解。SSH 作为 Linux 系统管理的核心工具，其广泛的适用性（远程登录、文件传输、端口转发）让我认识到了网络安全管理的基础知识。在今后的工作中，我会重视 SSH 密钥管理，定期更新和维护。【课程大纲-思政融合点4】
